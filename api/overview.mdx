The Trustee project exposes three API surfaces:

- **KBS HTTP API** вЂ” OpenAPI 3.1 compliant REST API for attestation and secret delivery
- **Attestation Service gRPC API** вЂ” gRPC service for evidence evaluation and policy management
- **Reference Value Provider gRPC API** вЂ” gRPC service for managing reference values used in attestation

## KBS HTTP API

### Base URL

```
http://<kbs>/kbs/v0
```

The metrics endpoint is served without the `/kbs/v0` prefix:

```
http://<kbs>/metrics
```

<Note>
  Deployments should use HTTPS in production. The KBS uses TLS to allow clients to authenticate the server identity before submitting attestation evidence. See [HTTPS/TLS configuration](/kbs/https-tls) for setup instructions.
</Note>

### Authentication

The KBS API has two authentication modes depending on the endpoint category:

| Category | Auth mechanism |
|---|---|
| Attestation protocol endpoints | Session cookie (`kbs-session-id`) set during `/auth`, or Bearer token from a prior attestation |
| Admin endpoints | Admin JWT signed with the operator's private key, passed as `Authorization: Bearer <token>` |

### Endpoint categories

#### Attestation protocol endpoints

These endpoints implement the **Request-Challenge-Attestation-Response (RCAR)** protocol. A client (Key Broker Client) running inside a confidential guest uses these to authenticate itself and retrieve secrets.

| Method | Path | Description |
|---|---|---|
| `POST` | `/kbs/v0/auth` | Initiate attestation вЂ” send a `Request` payload, receive a `Challenge` and session cookie |
| `POST` | `/kbs/v0/attest` | Submit TEE evidence вЂ” receive an `AttestationToken` JWT on success |
| `GET` | `/kbs/v0/resource/{repository}/{type}/{tag}` | Retrieve an encrypted secret resource (requires session cookie or Bearer token) |

#### Admin endpoints

These endpoints are for operators to configure the KBS. All require a valid admin JWT.

| Method | Path | Description |
|---|---|---|
| `POST` | `/kbs/v0/resource/{repository}/{type}/{tag}` | Register a secret resource |
| `POST` | `/kbs/v0/attestation-policy` | Set an attestation verification policy |
| `POST` | `/kbs/v0/resource-policy` | Set the resource distribution policy |
| `GET` | `/kbs/v0/resource-policy` | Get the current resource distribution policy |
| `POST` | `/kbs/v0/reference-value` | Register a reference value |
| `GET` | `/kbs/v0/reference-value/{reference_value_id}` | Query a reference value by ID |
| `GET` | `/metrics` | Prometheus metrics (no auth required) |

### Error format

All error responses use the [RFC 7807 Problem Details](https://www.rfc-editor.org/rfc/rfc7807) format:

```json
{
  "type": "https://github.com/confidential-containers/kbs/errors/<problem-type>",
  "detail": "Human-readable description of the error"
}
```

### RCAR protocol flow

The attestation protocol follows four steps:

```
Attester                                          KBS
   |                                               |
   |в”Ђв”Ђв”Ђв”Ђ POST /kbs/v0/auth (Request) в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–¶|
   |в—Ђв”Ђв”Ђв”Ђ 200 Challenge + Set-Cookie в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ|
   |                                               |
   |в”Ђв”Ђв”Ђв”Ђ POST /kbs/v0/attest (Attestation) в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ|
   |     Cookie: kbs-session-id=<id>               |
   |в—Ђв”Ђв”Ђв”Ђ 200 AttestationToken + Set-Cookie в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ|
   |                                               |
   |в”Ђв”Ђв”Ђв”Ђ GET /kbs/v0/resource/<repo>/<type>/<tag> в–¶|
   |     Cookie: kbs-session-id=<id>               |
   |в—Ђв”Ђв”Ђв”Ђ 200 JWE encrypted resource в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ|
```

After a successful attestation, the session cookie can be reused for subsequent resource requests within its validity period. Alternatively, the `AttestationToken` JWT returned by `/attest` can be used as a Bearer token for resource requests without a session cookie.
