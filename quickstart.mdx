---
title: "Quick start"
description: "Build and deploy the KBS locally on Ubuntu 22.04, upload a resource, and retrieve it using the kbs-client tool."
---

This guide walks you through deploying KBS in **Background Check mode** on Ubuntu 22.04 without containers. By the end you will have a running KBS, a resource stored in it, and a successful retrieval using the `kbs-client` CLI.

<Info>
If you prefer a container-based setup, see [Docker Compose deployment](/deployment/docker-compose) for a full Trustee cluster with a single command.
</Info>

<Steps>

<Step title="Install Rust">

Install the Rust toolchain using `rustup`:

```bash
curl https://sh.rustup.rs -sSf | sh
source "$HOME/.cargo/env"
```

</Step>

<Step title="Install system dependencies">

Add the Intel SGX repository and install build dependencies:

```bash
curl -L "https://download.01.org/intel-sgx/sgx_repo/ubuntu/intel-sgx-deb.key" | sudo gpg --dearmor --output /usr/share/keyrings/intel-sgx.gpg
echo "deb [arch=amd64 signed-by=/usr/share/keyrings/intel-sgx.gpg] https://download.01.org/intel-sgx/sgx_repo/ubuntu jammy main" \
	| sudo tee /etc/apt/sources.list.d/intel-sgx.list
sudo apt-get update
sudo apt-get install -y \
	build-essential \
	clang \
	libsgx-dcap-quote-verify-dev \
	libsgx-dcap-quote-verify \
	libtss2-dev \
	openssl \
	pkg-config \
	protobuf-compiler
```

</Step>

<Step title="Generate authentication keys">

The KBS uses an Ed25519 key pair to authenticate administrative requests. Generate the key pair into the `config/` directory:

```bash
openssl genpkey -algorithm ed25519 > config/private.key
openssl pkey -in config/private.key -pubout -out config/public.pub
```

<Note>
Keep `config/private.key` secret. It is used to authenticate all `kbs-client config` commands that modify the KBS, including uploading resources and updating policies.
</Note>

</Step>

<Step title="Build and install the KBS">

Build KBS in Background Check mode and install both the server and the client tool:

```bash
make background-check-kbs
sudo make install-kbs
make cli
sudo make install-cli
```

These commands must be run from the `kbs/` directory of the Trustee repository.

</Step>

<Step title="Start the KBS">

Start the KBS using the bundled configuration file:

```bash
sudo kbs --config-file config/kbs-config.toml &
```

The KBS listens on `http://127.0.0.1:8080` by default.

<Tip>
To enable HTTPS, pass `--private-key` and `--certificate` with the paths to your server-side PEM key and certificate. See [HTTPS/TLS configuration](/kbs/https-tls) for details.
</Tip>

</Step>

<Step title="Upload a resource">

Create a dummy resource file and upload it to the KBS using `kbs-client`:

```bash
cat > test/dummy_data << EOF
1234567890abcde
EOF

kbs-client --url http://127.0.0.1:8080 \
  config \
  --auth-private-key config/private.key \
  set-resource \
  --resource-file test/dummy_data \
  --path default/test/dummy
```

Resources are addressed by a three-part path: `<repository>/<type>/<tag>`. Here the resource is stored at `default/test/dummy`.

</Step>

<Step title="Retrieve the resource">

Request the resource from the KBS:

```bash
kbs-client --url http://127.0.0.1:8080 get-resource --path default/test/dummy
```

<Warning>
If you run the client outside a real TEE, the `sample` attester is used and the KBS rejects the evidence by default. To allow sample evidence for testing, update the resource policy to a permissive one:

```bash
kbs-client --url http://127.0.0.1:8080 \
  config \
  --auth-private-key config/private.key \
  set-resource-policy \
  --policy-file sample_policies/allow_all.rego
```

Do not use an allow-all policy in production.
</Warning>

</Step>

</Steps>

## Passport mode

The quickstart above uses Background Check mode, where a single KBS handles both attestation and resource delivery. Passport mode decouples these responsibilities into two separate KBS instances.

<Steps>

<Step title="Build and start the issuer KBS">

The issuer KBS verifies attestation evidence and issues tokens:

```bash
make passport-issuer-kbs
make install-issuer-kbs
issuer-kbs --socket 127.0.0.1:50001 --insecure-http --auth-public-key config/public.pub
```

</Step>

<Step title="Build and start the resource KBS">

The resource KBS delivers secrets to clients that present a valid attestation token:

```bash
make passport-resource-kbs
make install-resource-kbs
resource-kbs --socket 127.0.0.1:50002 --insecure-http --auth-public-key config/public.pub
```

</Step>

<Step title="Upload a resource to the resource KBS">

```bash
cat > test/dummy_data << EOF
1234567890abcde
EOF

kbs-client --url http://127.0.0.1:50002 \
  config \
  --auth-private-key config/private.key \
  set-resource \
  --resource-file test/dummy_data \
  --path default/test/dummy
```

</Step>

<Step title="Generate a TEE key pair">

The TEE key pair is used to bind the attestation token to the requesting TEE:

```bash
openssl genrsa -traditional -out test/tee_key.pem 2048
openssl rsa -in test/tee_key.pem -pubout -out test/tee_pubkey.pem
```

</Step>

<Step title="Obtain an attestation token from the issuer KBS">

```bash
kbs-client --url http://127.0.0.1:50001 attest \
  --tee-key-file test/tee_key.pem \
  > test/attestation_token
```

The public part of the TEE key is embedded in the token claims to identify the TEE.

</Step>

<Step title="Retrieve the resource using the attestation token">

```bash
kbs-client --url http://127.0.0.1:50002 get-resource \
  --attestation-token test/attestation_token \
  --tee-key-file test/tee_key.pem \
  --path default/test/dummy
```

</Step>

</Steps>

See [Attestation modes](/kbs/attestation-modes) for a detailed explanation of when to use Passport mode.
