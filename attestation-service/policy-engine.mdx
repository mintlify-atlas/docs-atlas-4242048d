title: "Policy engine"

The Attestation Service includes a flexible policy engine based on [Open Policy Agent (OPA)](https://www.openpolicyagent.org/). Policies are written in the Rego policy language and evaluated against the TCB claims extracted by the verifier driver.

Policy evaluation results are reflected in the [attestation token](/attestation-service/attestation-token). Each policy produces trust claim values for the eight AR4SI dimensions.

<Note>
Only EAR tokens are supported. Simple tokens have been removed. If you were using Simple tokens previously, update the `attestation_token_broker` section of your configuration file and rewrite your policies using the EAR format described below.
</Note>

## Policy selection

The `policy_ids` field in the attestation request specifies which policies to evaluate. If `policy_ids` is not provided, the AS uses the built-in `"default"` policy base — the AS then appends the TEE class suffix automatically (for example, `"default_cpu"` for CPU evidence, `"default_gpu"` for GPU evidence).

Multiple policies can be evaluated against the same evidence simultaneously:

```json
{
    "verification_requests": [ ... ],
    "policy_ids": ["default", "custom-policy-1"]
}
```

### Naming convention

When using EAR tokens, policies for CPU evidence must end in `_cpu`. The AS appends the `_cpu` suffix to the first `policy_ids` entry automatically when evaluating CPU evidence. If you upload a custom policy for CPU attestation, the policy ID must use this suffix (for example, `my-org_cpu`).

The built-in default policies are pre-loaded with these IDs:

| Policy ID | Device class |
|---|---|
| `default_cpu` | CPU (TDX, SGX, SNP, etc.) |
| `default_gpu` | NVIDIA GPU |
| `default_switch` | Network switch attestation |
| `default_ppcie` | PCIe device attestation |

For GPU attestation, use a corresponding policy ending in `_gpu` (for example, `default_gpu`). A separate policy is required for each device class being attested.

## Policy structure

A valid EAR policy must evaluate at least one of the following trust claim dimensions as an integer between -127 and 127:

- `instance_identity`
- `configuration`
- `executables`
- `file_system`
- `hardware`
- `runtime_opaque`
- `storage_opaque`
- `sourced_data`

The policy receives the TCB claims as `input`. These are the structured JSON claims generated by the verifier driver for the attested platform.

### Default policy

The AS ships with a default policy that handles multiple TEE platforms. If no policy is uploaded, this policy is used:

```rego
package policy

import rego.v1

# Default trust claim values (used when no platform-specific rule matches)
default executables := 33   # executables not recognized
default hardware := 97      # hardware not recognized but should be
default configuration := 36 # configuration unavailable
default file_system := 0    # no assertion
default instance_identity := 0
default runtime_opaque := 0
default storage_opaque := 0
default sourced_data := 0

trust_claims := {
    "executables": executables,
    "hardware": hardware,
    "configuration": configuration,
    "file-system": file_system,
    "instance-identity": instance_identity,
    "runtime-opaque": runtime_opaque,
    "storage-opaque": storage_opaque,
    "sourced-data": sourced_data,
}
```

Platform-specific rules override the defaults when their conditions are satisfied.

## Writing policies

### SGX policy

Check that the enclave has the expected `mr_enclave` and `mr_signer` measurements:

```rego
package policy

import rego.v1

default executables := 33

executables := 3 if {
    input["sgx.body.mr_enclave"] == "8f173e4613ff05c52aaf04162d234edae8c9977eae47eb2299ae16a553011c68"
    input["sgx.body.mr_signer"] == "83d719e77deaca1470f6baf62a4d774303c899db69020f9c70ee1dfc08c7ce9e"
}
```

### TDX policy

Check the TDX module, guest firmware, TCB status, and collateral expiration:

```rego
package policy

import rego.v1

default executables := 33

executables := 3 if {
    input["tdx.quote.body.mr_td"] == "705ee9381b8633a9fbe532b52345e8433343d2868959f57889d84ca377c395b689cac1599ccea1b7d420483a9ce5f031"
    input["tdx.quote.body.mr_seam"] == "2fd279c16164a93dd5bf373d834328d46008c2b693af9ebb865b08b2ced320c9a89b4869a9fab60fbe9d0c5a5363c656"
    input["tdx.ccel.kernel"] == "5b7aa6572f649714ff00b6a2b9170516a068fd1a0ba72aa8de27574131d454e6396d3bfa1727d9baf421618a942977fa"
}
```

### IBM SE policy

Check SE attestation fields including version, user data, and host key hashes:

```rego
package policy

import rego.v1

default executables := 33

converted_version := sprintf("%v", [input["se.version"]])

executables := 3 if {
    converted_version == "256"
    input["se.user_data"] == "00"
    input["se.tag"] == "773780962a7350165054673b6c54235d"
    input["se.image_phkh"] == "92d0aff6eb86719b6b1ea0cb98d2c99ff2ec693df3efff2158f54112f6961508"
    input["se.attestation_phkh"] == "92d0aff6eb86719b6b1ea0cb98d2c99ff2ec693df3efff2158f54112f6961508"
}
```

### SNP policy with reference values

Use the `query_reference_value()` function to check claims against values stored in RVPS:

```rego
package policy

import rego.v1

default executables := 33
default hardware := 97
default configuration := 36

executables := 3 if {
    input.snp
    input.snp.measurement in query_reference_value("snp_launch_measurement")
}

hardware := 2 if {
    input.snp
    input.snp.reported_tcb_bootloader in query_reference_value("snp_bootloader")
    input.snp.reported_tcb_microcode in query_reference_value("snp_microcode")
    input.snp.reported_tcb_snp in query_reference_value("snp_snp_svn")
    input.snp.reported_tcb_tee in query_reference_value("snp_tee_svn")
}

configuration := 2 if {
    input.snp
    input.snp.policy_debug_allowed == false
    input.snp.policy_migrate_ma == false
    input.snp.platform_smt_enabled == query_reference_value("snp_smt_enabled")
    input.snp.platform_tsme_enabled == query_reference_value("snp_tsme_enabled")
}
```

### TPM with reference values

```rego
package policy

import rego.v1

hardware := 2 if {
    input.tpm
}

executables := 3 if {
    input.tpm
    input.tpm.pcr11 in query_reference_value("tpm_pcr11")
}

configuration := 0 if {
    input.tpm
}
```

## The `query_reference_value()` function

Policies can call `query_reference_value(key_name)` to retrieve reference values stored in the [RVPS](/rvps/overview). This function returns the value associated with `key_name` in the RVPS store, or `null` if no value is found.

```rego
# Check a measurement against a set of accepted reference values
executables := 3 if {
    input.tpm
    input.tpm.pcr11 in query_reference_value("pcr11")
}
```

Reference values are typically sets (arrays), and the `in` operator checks for membership. Scalar equality is also valid for single-value reference values.

## Policy extensions

A policy can optionally define an `extensions` claim. Extensions are added to the attestation token. Each extension is an object with the following fields:

```json
{
    "name": "<collision-resistant-public-claim-name>",
    "key": <i32>,
    "value": <any valid JSON>
}
```

- If the extension is registered in the EAR specification, `key` is a positive integer.
- For private extensions, `key` must be negative.

The default policy defines the `ear.trustee.identifiers` extension (key `-18`):

```rego
extensions := [
    {"name": "ear.trustee.identifiers",
         "key": -18,
         "value": {
            "validated": validated_identifiers
        }
    }
]

validated_identifiers := object.union_n([
    container_images_id,
    container_uids_id,
])

container_images := [img |
    container := input["init_data_claims"]["agent_policy_claims"]["containers"][_]
    img := container["OCI"]["Annotations"]["io.kubernetes.cri.image-name"]
]
```

## Uploading policies

<Tabs>
  <Tab title="REST API">
    Upload a policy by POSTing to the `/policy` endpoint. The policy content must be base64-encoded (URL-safe, no padding):

    ```bash
    curl -k -X POST http://127.0.0.1:8080/policy \
         -H 'Content-Type: application/json' \
         -d '{
               "type": "rego",
               "policy_id": "my-policy_cpu",
               "policy": "<base64-url-safe-no-pad-encoded-rego-content>"
             }'
    ```

    The `/policy` endpoint request payload fields:

    | Field | Type | Description |
    |---|---|---|
    | `type` | string | Policy type — use `"rego"` |
    | `policy_id` | string | Identifier for the policy (must end in `_cpu` for CPU policies) |
    | `policy` | string | Base64 URL-safe no-padding encoded Rego policy content |

    Then reference the uploaded policy in an attestation request:

    ```bash
    curl -k -X POST http://127.0.0.1:8080/attestation \
         -H 'Content-Type: application/json' \
         -d '{
               "verification_requests": [ ... ],
               "policy_ids": ["my-policy_cpu"]
             }'
    ```
  </Tab>
  <Tab title="gRPC">
    Use the `SetAttestationPolicy` RPC defined in `attestation.proto`:

    ```bash
    REQ=$(cat tests/coco-as/policy/grpc-set-policy.json)
    grpcurl \
      -plaintext \
      -import-path ../protos \
      -proto ../protos/attestation.proto \
      -d @ 127.0.0.1:50004 attestation.AttestationService/SetAttestationPolicy <<EOF
    $REQ
    EOF
    ```

    The request payload follows the same structure as the REST API. The policy content must be base64 URL-safe no-padding encoded.

    Then evaluate evidence against the policy using `AttestationEvaluate`:

    ```bash
    REQ=$(cat tests/coco-as/policy/check.json)
    grpcurl \
      -plaintext \
      -import-path ../protos \
      -proto ../protos/attestation.proto \
      -d @ 127.0.0.1:50004 attestation.AttestationService/AttestationEvaluate <<EOF
    $REQ
    EOF
    ```
  </Tab>
</Tabs>
